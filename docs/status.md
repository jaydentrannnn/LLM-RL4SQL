## Project Summary
Our project will be using Reinforcement Learning to finetune a large language model in the text2sql task. To explore what affects final model performance we will be evaluating various reward designs as well as system prompts. In addition, we will be exploring how our model with its limited compute compares to other models that are more expensive to run. We hope that the results of this project could help with developing resource constrained LLMs to have strong performance within specific tasks.


## Approach

The algorithm that we are using is called Group Relative Policy Optimization (GRPO). This RL algorithm has gained popularity for the LLM fine-tuning task for many practical reasons that are applicable to our problem. One such reason is that it does not use a critic model to score the responses generated by the LLM. This reduces memory use, which is important to our project given our compute resources, and stabilizes training, as learning a value function can be difficult in the LLM domain. Additionally, GRPO utilizes the fact that within LLM fine-tuning, final outputs are evaluated and rewarded rather than rewarding individual tokens. This can be seen in the definition of the advantage, where completions within a group are ranked against each other to provide a signal for the model on which compilations are favorable and which are not.

$\hat{A}_{i,t} = \frac{r_i - \mathrm{mean}(\bm{r})}{\mathrm{std}(\bm{r})}$

Where $r_i$ is the reward for the $i$-th completion and $\mathrm{mean}(\bm{r})$ and $\mathrm{std}(\bm{r})$ are statistics calculated from the group of completions. 

Additionally, the GRPO loss includes a Kullback Leiber (KL) divergence term. KL is a measure of how similar two probability distributions are. By measuring the KL divergence between the policy that is being learned and the reference policy, there is an incentive to not learn policies that stray away from a stable reference policy. Although the true KL divergence can be difficult to estimate, the estimate we used is defined below:

$\mathbb{D}_{\mathrm{KL}}[\pi_\theta ||\pi_{\mathrm{ref}}] = \frac{\pi_{\mathrm{ref}}(o_i, t | q, o_{i, < t})}{\pi_{\theta}(o_i, t | q, o_{i, < t})} - \log\frac{\pi_{\mathrm{ref}}(o_i, t | q, o_{i, < t})}{\pi_{\theta}(o_i, t | q, o_{i, < t})} - 1$

Where $o_{i, t}$ and $o_{i,<t}$ for completion $i$ is the output token at index $t$ and all output tokens before index $t$ respectively. $q$ is the query token string.

The final clipped surrogate objective is defined below:

$\mathscr{L}_{\mathrm{GRPO}}(\theta)=-\frac{1}{\sum_{i=1}^G |o_i|}\sum_{i=1}^G \sum_{t=1}^{|o_i|}[\mathrm{min}(\frac{\pi_{\theta}(o_i, t | q, o_{i, < t})}{\pi_{\theta_{\mathrm{old}}}(o_i, t | q, o_{i, < t})}\hat A_{i, t}, \mathrm{clip}(\frac{\pi_{\theta}(o_i, t | q, o_{i, < t})}{\pi_{\theta_{\mathrm{old}}}(o_i, t | q, o_{i, < t})}, 1 - \epsilon, 1 + \epsilon)\hat A_{i, t}) -\beta\mathbb{D}_{\mathrm{KL}}[\pi_\theta||\pi_{\mathrm{ref}}]]$

Where clipping is used to ensure that updates do not deviate excessively from the reference policy.

We have been using the [Spider](https://yale-lily.github.io/spider) dataset in order to train and evaluate our model. The dataset contains manually generated text descriptions of various SQL queries on various databases. The dataset organizes this information in a JSON format and contains additional information about the schema of each database as well as the actual databases themselves. In order to provide initial context within our queries and information for scoring the responses, we augmented the dataset to include two additional columns for this information. This was done by defining a schema column that contains textual information about the schema and a result column that contains the results of running the queries on their corresponding databases.

In order to achieve optimal performance, creating the correct system and user prompts is very important. Currently we are using the following prompts but intend to explore various others to see their effects on performance.

### System Prompt
```
You are a SQL expert. Output only a single valid and executable SQL in <sql> tags.
```

### User Prompt
```
For the SQL database with the following schema:
{table_info}
Create a query for the question: {question}
```

Reward Information

Syntax check reward function—To validate that the generated query is syntactically valid, we design a reward function that executes the query on the corresponding table and sees if it executes correctly without yielding any error. Specifically, the function gives 1.0 for queries that don’t yield any error and 0.0 for queries that yield any SQLite Error or Exception.
Schema linking reward function—The accuracy in translating plain text into schema items is critical in the Text-to-SQL model since it must correctly identify and reference the correct tables and columns. For our reward function, we extract all the schema items from the generated query and the provided gold query using SQLGlot library and return the Jaccard Similarity score as the reward.
N-Gram similarity reward function—Utilizing the structural structure of SQL query, we built a reward function that works based on this structure. Specifically, we first tokenize the generated SQL query and the golden query into lists of tokens (order matters) using SQLGlot library. We then utilize the SequenceMatcher algorithm from Difflib library to give a score (0.0-1.0) that represents the difference between the two sequences and return that as the reward for the model.

In order to facilitate training we have been utilizing the [GRPO Trainer](https://huggingface.co/docs/trl/grpo_trainer) from HuggingFace. In regards to hyperparameters we have been using the default ones found in the [GRPOConfig](https://huggingface.co/docs/trl/v0.28.0/en/grpo_trainer#trl.GRPOConfig).

## Evaluation

To evaluate the quality of our GRPO fine-tuned model on the text-to-SQL task, we will primarily use execution accuracy on the Spider 1.0 test set. Execution accuracy measures whether the SQL query generated by the model, when executed against the corresponding data, produces the same result set as the ground truth query. We will also pay attention to the Jaccard Similarity between the two sets of results since it illustrates how concise the generated query is. This metric is preferred over exact string match because it correctly credits semantically equivalent queries that may differ in syntax, column ordering, or formatting. We will use the official Spider evaluation script alongside the provided SQLite database files to compute this metric. Our test set consists of the standard Spider 1.0 held-out split, which contains queries across databases and domains not seen during training, allowing us to assess the model’s generalization ability. Besides, we might test the model on other datasets like BIRD if time permitted.

To contextualize our results, we will compare execution accuracy against a baseline of the un-fine-tuned Qwen 3.0-0.6B model evaluated on the same test set under identical inference conditions. This allows us to directly quantify the improvement attributable to GRPO training. Other than that, we also wanted to compare the running time and accuracy against existing pipelines for the same task.

## Remaining Goals and Challenges
Our goal at a high level is to improve our current text-to-SQL pipeline into a better evaluated system that shows clear improvement over a baseline model on Spider. We plan to continue iterating on reward design and test different preprocessing strategies that improve results. We will continue comparing how different base models respond to the same GRPO setup. We also want to expand our evaluation with multiple concurrent evaluations execution accuracy and exact match, to get more details about helpful changes.  

One of the challenges is that when reward tuning, a query can be logically correct but look different from the reference, being marked wrong. This is an issue specific to query scoring that we hope to fix with more advanced evaluation scoring reward functions. We also face practical constraints around compute, training time, and debugging to slow iteration, especially as we test multiple reward functions and model variants.
